Technology Choices When Building Your Own Blockchain

You want to create your own cryptocurrency but your options are dismally limited. If you’re new to blockchain development and are weighing the options about which path to take to bootstrap your token, you have just about three good ones: 1) fork an existing codebase like Bitcoin’s, 2) write a smart contract under a token standard on Ethereum, or 3) use a developer-friendly blockchain framework like the Cosmos-SDK.

While forking the battle-hardened codebase of a protocol that hasn’t been broken in the history of its existence appears to be erring on the safe side, it’s far from being your best option. In this blog post, you’ll develop the understanding of the benefits and drawbacks of creating tokens from option 1) forking an existing codebase and option 3) using a blockchain framework like the Cosmos-SDK. For this post, option 2) is out of scope. A more comprehensive deep dive in a future post about the various platforms, tools, and tradeoffs of each can be written contingent upon demand.

Forking bitcoind (Bitcoin Core daemon)

Forking an existing codebase like that of Bitcoin’s may be the simplest option at the outset, but it has many drawbacks that are guaranteed to make your developer tear their hairs out maintaining your forked codebase.

Drawbacks

  • Technical debt. Bitcoin Core developers make bug fixes, patches, and upgrades via bandages, leading to legacy code riddled with technical debt. Navigating this terrain can take as much effort as writing your codebase from scratch.
  • Unmaintainable codebase. Bitcoind is still in the midst of a substantial refactor and cleanup of the initial somewhat deficient codebase.
    This results in frequent large changes that touch many parts of the codebase and puts substantial pressure on downstream maintainers. It takes a substantial effort for even seasoned C++ developers to familiarize themselves with the Bitcoin reference client written in C++.

Shortlist of Properties

  • UTXOs. The Unspent Transaction Output (UTXO) model is optimized for transaction validation in Bitcoin and is particularly specialized for payments. The innovation behind this model introduced the commutativity of transactions; the order of transactions is insignificant. This helps with better user experience under reorganizations (a consequence of Bitcoin’s decentralized nature) but is not an ideal model in other contexts. UTXOs allow you to do more parallel smart contract evaluation/verification. As such, it is only appropriate to model your application after this if you explicitly intend to. In addition, it is easier to apply privacy features like confidential transactions and a zero-knowledge protocol under a UTXO model. Cosmos and Ethereum are account-based systems.
  • Proof-of-Work. Bitcoin is deeply embedded in Proof-of-Work. Again, it is only appropriate to mirror the security model of your application after Bitcoin’s if you explicitly intend to. In a permissioned setting, Proof-of-Work (PoW) carries substantially greater overhead than security models such as Proof-of-Authority (PoA) or Proof-of-Stake (PoS). Tendermint BFT has been and continues to be stress-tested in enterprise today.

Alternative Bitcoin Implementations

The two alternative implementations of the Bitcoin reference client that have mined blocks on the live network are:

  • bcoin—alternative Bitcoin implementation written in node.js; authored by Fedor Indutny & lead maintainer, Christopher Jeffrey
  • btcd—alternative Bitcoin implementation written in Go; authored by Dave Collins; forked and used in production by the Lightning Network protocol (lnd)

If you want to build your own application from an alternative implementation like btcd or bcoin, the same drawbacks hold. Changing an existing stack to meet your specific user requirements is significantly harder than starting from a solid base layer framework.

Case Study—Zcash

Zcash in 2016 forked bitcoind to become its own altchain (alternative chain forked from Bitcoin). It had all the bugs of bitcoind, but unfortunately, upstream fixes from Bitcoin Core were difficult to maintain in Zcash. Since the forked codebase had been changed with all the privacy enhancements and other features, upstream changes weren’t able to be merged. The Zcash team has occasionally voiced regret for the challenges posed as a fork of bitcoind.

Shortlist of Properties

  • Accounts. Account-based systems have much simpler client code, making them easier to tinker with. Frequently, you need to tie up less collateral in smart contracts.
  • Developability; language agnostic. Allows your developers to build your blockchain by programming in the high-level business logic alone in their preferred programming language; the low-level replicated state machine and gossip layer protocols are taken care of for you with Tendermint.
  • Composable modules. The Cosmos-SDK is packaged with batteries included, offering you a set of universal modules that you can simply plug into your application-specific blockchain. You get features like tokens, governance, interoperability, and staking pre-built. Forking an existing codebase necessarily means that you would have to write those functionalities from scratch yourself.
  • InterBlockchain Communication (IBC). Interoperating with other blockchains in Cosmos is as easy as plugging in the IBC module pre-packaged in the SDK. With IBC, your cryptocurrency can actually access any other blockchain that exists in the Cosmos ecosystem, enabling a gigantic liquidity pool—a major enhancing factor for any cryptocurrency.
  • Blockchain framework. You’ll use a relatively simple data flow model for the application logic, allowing you to build your middleware stack. It is analogous to early web server frameworks such as Express, written in JavaScript.
  • Swappable consensus. With the Cosmos-SDK, you have the full flexibility of deploying a custom-tailored consensus algorithm should you choose not to use Tendermint. Whether you desire to use a PoA or PoS-like construct; the choice is yours. The Cosmos SDK supports Proof-of-Stake or any other permission mechanism like election by other validators or approval from some authority.
  • Public or permissioned setting. If you want to build a custom permissioned public blockchain, for example, your only options are Stellar and Cosmos-SDK. The Cosmos-SDK is far more flexible than Stellar, which is only designed for payments/value transfer.

Tendermint BFT

Tendermint is the first production-ready Byzantine Fault Tolerant Proof-of-Stake consensus algorithm in the blockchain ecosystem. With it comes myriad properties one would want included in a blockchain. The most prominent public blockchain network entering the marketplace in the coming weeks built on top of Tendermint will be the Cosmos Network.

Shortlist of Properties

  • Security. A Byzantine Fault Tolerance-based PoS chain gives you fault-tolerance guarantees of up to any fault f < n/3 , where n is the number of distributed consensus nodes in your system responsible for block creation.
  • Speed and scalability. Byzantine Fault Tolerant consensus protocols entail optimistic block creation since we don’t need a network propagation safety margin due to the fact that a Tendermint chain never forks. A Proof-of-Work-based chain is akin to operating a BBS/Dialup while the world moves to HTML/HTTP/TCP/IP/DSL with Proof-of-Stake.
  • Safety. There is no chance for the network to go split brain unless a significant portion of the network is malicious and colluding. A Tendermint chain doesn’t fork, so your chain doesn’t face the risk of reorganizations. This is a much better user experience, as users who send transactions through your network can reasonably expect those transactions not to get reverted.
  • Tailorable consensus. It is possible to deploy a Proof-of-Authority network using Tendermint BFT consensus. Generally, consensus (i.e. Tendermint, etc.), isn’t tied to security (i.e. PoA, PoW, PoET, PoS, etc.) in this model.

In Closing

It may seem daunting to consider every consensus protocol in the wild on top of picking a secure environment to build on. To start your decision-making process, ask yourself these key questions:

1. Decide on a platform: Bitcoin, Ethereum, Cosmos, etc. To make this decision, start with your user requirements and work backwards. Is it payments your users are transacting in? Is it sharing storage space? Is it perhaps something more niche, like IoT?
2. What does your network of trusted/untrusted players look like? Decide from that kickoff point to pick your most optimal security setting: PoA (least overhead, permissioned, trusted), PoS (medium overhead, permissionless, trustless), PoW (highest overhead, permissionless, trustless).
3. Which language is your development team most familiar/comfortable with? C++ (bitcoind), JavaScript (bcoin), Go (Cosmos/btcd), Rust (Cosmos), Solidity (Ethereum), Plutus (Cardano), or any other language (Tendermint ABCI), etc.?

Cosmos-SDK is the new kid on the block. Meticulously designed for ease-of-development, you’re now able to build a full blockchain—not an altchain or an ERC20—with as little development overhead as it takes to program an application.
